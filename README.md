# S.T.A.L.K.E.R. Mod Launcher

Windows-лаунчер профилей для локальной GOG/DRM-free версии **S.T.A.L.K.E.R.: Shadow of Chernobyl**.

Лаунчер не поддерживает Steam и не изменяет папку установленной игры. Папки модов также не изменяются.

## Возможности

- отдельные профили с собственными настройками запуска;
- обычные моды, которые накладываются поверх базовой игры;
- автономные моды с собственным движком и файлами игры;
- включение, отключение и изменение порядка модов;
- сортировка модов и профилей перетаскиванием;
- автоматическое определение подходящего EXE;
- пошаговый мастер создания профиля;
- предварительная проверка источников и итогового бинарника перед запуском;
- отдельные сохранения, настройки и логи для обычных профилей;
- проверка состояния профиля, управление кэшем workspace, поиск логов и crash dump;
- анализ наложения конфигураций, бинарных и остальных файлов модов;
- просмотр и копирование скриншотов;
- импорт и экспорт профилей;
- учет игрового времени.

## Требования

- Windows 10 или Windows 11 x64;
- локальная GOG/DRM-free установка Shadow of Chernobyl для обычных профилей;
- права на создание файлов в папке workspace;
- для сборки исходного кода: .NET 8 SDK.

Готовая self-contained сборка не требует отдельно установленного .NET.

## Первый запуск обычного мода

1. Нажмите кнопку создания профиля.
2. В мастере выберите обычный профиль и задайте название.
3. Выберите корневую папку Shadow of Chernobyl и добавьте папки модов.
4. Проверьте автоматически найденный исполняемый файл.
5. Завершите мастер и нажмите **Запустить**.

Для оригинальной игры обычно используется `bin\XR_3DA.exe` или `bin\xr_3da.exe`. Моды на OGSR Engine часто используют `bin_x64\xrEngine.exe`.

## Автономный мод

Автономный мод уже содержит собственный движок и необходимые игровые файлы.

1. Создайте профиль.
2. Включите переключатель **Автономный мод**.
3. Добавьте папку автономного мода.
4. Выберите его EXE, если он не был найден автоматически.
5. Нажмите **Запустить**.

Автономный профиль запускается непосредственно из папки мода. Workspace для него не создается. Место хранения сохранений и логов определяет сам автономный мод.

## Порядок модов

Моды применяются сверху вниз. **Нижний включенный мод имеет больший приоритет** и заменяет совпадающие файлы модов выше.

Порядок можно изменить:

- перетаскиванием строки мышью;
- кнопками **Выше** и **Ниже**.

После изменения порядка подпись workspace меняется. При следующем запуске launcher пересобирает workspace в новом порядке.
Подписи под модами отдельно показывают количество заменённых конфигураций и бинарных файлов,
а также мод, который предоставляет итоговый запускаемый EXE. Обычная замена файлов патчем
считается ожидаемым наложением, а не критической ошибкой.

## Как работает workspace

Для обычного профиля launcher создает управляемую папку workspace вне установленной игры:

```text
<диск игры>\StalkerModLauncher\Workspaces\<имя профиля>-<короткий ID>\
  current\
  userdata\
  build-manifest.json
```

- `current` содержит объединенное представление игры и модов;
- `userdata` содержит сохранения, пользовательские настройки и логи профиля;
- `build-manifest.json` хранит подпись последней успешной сборки.

Launcher использует строгий экономный режим:

- файлы на одном NTFS-диске подключаются через hard link;
- файлы с других дисков подключаются через symbolic link;
- обычная копия в `current` создается только для `fsgame.ltx`, который launcher переписывает для изоляции профиля;
- если hard link и symbolic link создать нельзя, сборка останавливается с понятной ошибкой. Launcher не начинает скрыто копировать гигабайты.

Для symbolic link между разными дисками включите **Режим разработчика** Windows или запустите launcher от имени администратора.
Сохранения, пользовательские настройки, логи и скриншоты создаются отдельно в `<workspace профиля>\userdata` и не являются ссылками на данные установленной игры.

Проводник может показывать большой размер workspace, потому что учитывает полный размер hard-linked файлов. Это не означает, что каждый такой файл занял место на диске повторно.

## Защита файлов

- launcher не пишет в папку установленной игры;
- launcher не удаляет папки модов;
- удаление обычного профиля разрешено только для workspace внутри управляемой папки launcher и при наличии защитного marker-файла;
- удаление автономного профиля не удаляет папку автономного мода;
- настройки записываются атомарно в JSON и имеют резервную копию;
- одновременно может работать только один экземпляр launcher, чтобы два окна не перезаписали настройки друг друга;
- отмененная или неудачная сборка не получает manifest успешного workspace.

## Где хранятся данные

Настройки launcher:

```text
%AppData%\StalkerModLauncher\settings.json
%AppData%\StalkerModLauncher\settings.backup.json
```

`settings.json` содержит версию схемы, общие параметры интерфейса и профили. Поле
`LastBrowsedGamePath` запоминает только последнюю папку, открытую в диалоге выбора,
и никогда не используется как скрытая папка игры для профиля. Каждый обычный профиль
обязан хранить собственный `GameInstallPath`.

Пути к игре и модам хранятся абсолютными. После переноса файлов выберите новую папку
игры в профиле и заново укажите перемещенные папки модов. Launcher не заменяет пути
автоматически, потому что совпадающее имя папки не гарантирует, что это тот же мод.

Резервный корень workspace:

```text
%LocalAppData%\StalkerModLauncher\Workspaces
```

Если папка игры находится на другом диске, новые workspace создаются на диске игры, чтобы hard links могли экономить место.

### Перенос workspace

Не переносите workspace вручную. Откройте **Состояние** и нажмите **Перенести**.
Launcher создаст управляемую папку на выбранном диске, скопирует только ценный `userdata`,
переключит профиль на новый путь и затем удалит старый workspace. Каталог `current`
не копируется, потому что является восстанавливаемым кэшем.

Для резервного копирования достаточно сохранить `userdata`; папку `current` копировать
не требуется.

## Диагностика

Кнопка **Состояние** проверяет:

- путь к базовой игре;
- существование папок модов;
- выбранный EXE;
- workspace и его marker;
- сохранения;
- последний игровой лог;
- последний crash dump.

Там же отображаются логический и приблизительный фактический размер workspace,
количество hard link, symbolic link и локальных файлов. Кнопка **Очистить кэш**
удаляет только восстанавливаемую папку `current`; `userdata` с сохранениями,
настройками и логами остаётся на месте. Кнопка **Пересобрать** заново подготавливает
workspace без запуска игры. Кнопка **Перенести** безопасно копирует `userdata` в
новый workspace на выбранном диске; `current` не копируется и будет собран заново.

Статистика workspace сохраняется в `build-manifest.json` во время сборки, поэтому
открытие окна **Состояние** не выполняет долгий обход всех файлов. Для workspace,
созданного старой версией launcher, подробные размеры появятся после пересборки.

Непосредственно перед каждым запуском launcher дополнительно проверяет итоговый EXE,
источники файлов, `fsgame.ltx`, свободное место и возможность создавать symbolic link
для модов с других дисков. Ошибки останавливают запуск до изменения workspace.

Если профиль не запускается, сначала откройте **Состояние**, затем проверьте сообщения в **Журнале**.

## Совместимость с играми и движками X-Ray

Архитектура workspace не привязана к названию конкретной игры. Она работает с общей
для X-Ray и производных движков моделью: базовая папка, упакованные `gamedata.db*`,
распакованная папка `gamedata`, `fsgame.ltx`, исполняемый файл и последовательность
накладываемых папок модов.

Поэтому отдельные адаптеры для Shadow of Chernobyl, Clear Sky, Call of Pripyat,
Anomaly и OGSR сейчас не используются: они дублировали бы одну и ту же файловую
логику. Расширение разделено по реальным обязанностям: сканирование источников,
создание ссылок, поиск исполняемого файла, настройка пользовательских данных и
хранение manifest. Специальный адаптер потребуется только для игры или движка с
действительно другой структурой и правилами запуска.

## Скриншоты

Окно **Скриншоты** ищет изображения в стандартных каталогах профиля и автономного мода.

- ПКМ по миниатюре: **Скопировать в буфер обмена**;
- двойной щелчок ЛКМ по полноэкранному изображению: копировать текущий скриншот;
- колесо мыши или клавиши стрелок: предыдущий или следующий скриншот;
- `Esc`: выйти из полноэкранного просмотра.

Поддерживаются `.png`, `.jpg`, `.jpeg` и `.bmp`.

## Сборка из исходного кода

```powershell
dotnet build .\StalkerModLauncher.sln
dotnet test .\StalkerModLauncher.sln -c Release
dotnet run --project .\src\StalkerModLauncher\StalkerModLauncher.csproj
```

Публикация одного self-contained EXE для Windows x64:

```powershell
dotnet publish .\src\StalkerModLauncher\StalkerModLauncher.csproj `
  -c Release `
  -r win-x64 `
  --self-contained true `
  -p:PublishSingleFile=true `
  -p:IncludeNativeLibrariesForSelfExtract=true `
  -p:DebugType=none
```

## Структура проекта

```text
src/StalkerModLauncher/
  Models/          JSON-модели и состояние профилей
  ViewModels/      состояние и тематические группы команд интерфейса
  Views/           WPF-окна и переиспользуемые UserControl-компоненты
  Themes/          палитра и общие стили WPF
  Services/        workspace, запуск, хранение, диагностика и файловая логика
  Infrastructure/  базовые MVVM-компоненты

tests/StalkerModLauncher.Tests/
  модульные и интеграционные тесты
```

Основной стек: .NET 8, WPF, MVVM, JSON. Облачные сервисы и база данных не используются.
