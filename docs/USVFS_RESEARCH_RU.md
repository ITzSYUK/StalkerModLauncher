# Исследовательская ветка USVFS

[Русская версия](USVFS_RESEARCH_RU.md) | [English version](USVFS_RESEARCH_EN.md)

Эта ветка намеренно отделена от `main`. Стабильным режимом лаунчера остаётся `LinkedWorkspace`, а ветка предназначена для исследования интеграции с [ModOrganizer2/usvfs](https://github.com/ModOrganizer2/usvfs) без риска для существующего запуска через workspace.

## Что было проверено

`ModOrganizer2/usvfs` — нативная библиотека на C++, которая с помощью перехвата API создаёт виртуальные файловые ссылки, видимые только выбранным процессам. В публичном README её назначение описано как наложение нескольких каталогов на одну целевую директорию. Там же указаны лицензия GPLv3 и инструменты сборки: CMake, Python 3+ и vcpkg.

```powershell
cmake --preset vs2022-windows-x64
cmake --build --preset vs2022-windows-x64 --config Release
```

Публичный заголовочный файл предоставляет основные функции управления runtime:

- `usvfsCreateVFS` / `usvfsConnectVFS`;
- `usvfsVirtualLinkFile`;
- `usvfsVirtualLinkDirectoryStatic`;
- `usvfsCreateProcessHooked`;
- `usvfsCreateVFSDump`.

Такая архитектура подходит лаунчеру лучше удалённого самописного hook-эксперимента: managed-код формирует план наложения, а нативная библиотека отвечает за внедрение в процесс и перехват файловой системы.

## Локальная сборка и состояние PoC

`ModOrganizer2/usvfs` был локально собран для x64 и x86 с помощью Visual Studio Build Tools, CMake, Ninja и vcpkg. Исследовательские бинарные артефакты хранятся вне репозитория в `.external/` и временном каталоге исходников и сборки.

Ветка содержит небольшой внешний proof of concept в `research/usvfs-poc`. Он связывается с локальным `usvfs_x64.lib`, копирует `usvfs_x64.dll` и `usvfs_proxy_x64.exe` в выходную папку, после чего:

1. создаёт каталоги `base` и `mod`;
2. накладывает оба каталога на один виртуальный корень через `usvfsVirtualLinkDirectoryStatic`;
3. запускает дочерний процесс через `usvfsCreateProcessHooked`;
4. проверяет видимость уникальных файлов каждого слоя и приоритет `mod` над `base` при совпадении путей.

Подтверждённый результат:

```text
shared=mod
base-only=base
mod-only=mod
nested=mod-system
USVFS overlay PoC passed.
```

Следующий мост также проверен: `research/usvfs-managed-poc` использует managed-адаптер `UsvfsRuntime` со стороны лаунчера, загружает тот же официальный `usvfs_x64.dll` через P/Invoke и запускает hooked-процесс из C#.

Подтверждённый результат:

```text
shared=mod
base-only=base
mod-only=mod
nested=mod-system
Managed USVFS PoC passed.
```

Ветка содержит экспериментальный `UsvfsLaunchBackend`. Backend доступен, когда `usvfs_x64.dll` и `usvfs_proxy_x64.exe` находятся рядом с исполняемым файлом лаунчера. Переменная окружения `STALKER_MOD_LAUNCHER_ENABLE_OFFICIAL_USVFS=1` сохраняется для изолированных исследовательских запусков, но больше не требуется для подготовленной экспериментальной сборки. В настройках каждого неавтономного профиля можно выбрать `LinkedWorkspace` или `VirtualFileSystem`.

Интеграция поддерживает обе архитектуры EXE. x64-цели используют managed-адаптер и `usvfs_x64.dll` напрямую. x86-цели запускаются небольшим helper-процессом соответствующей разрядности `StalkerModLauncher.UsvfsX86Host.exe`, который загружает `usvfs_x86.dll`, применяет тот же план наложения и ожидает завершения игрового процесса. Это намеренно исключает нестабильный cross-bitness proxy-путь, обнаруженный во время исследования. Лаунчер разрешает одновременно запускать только один USVFS-профиль независимо от архитектуры; x86-сессия работает в отдельном helper-процессе.

Helper остаётся активным, пока работают внедрённые дочерние процессы. Благодаря этому короткоживущий launcher, например Gunslinger `Play.exe`, не уничтожает USVFS-сессию до завершения дочернего `xrEngine.exe`.

Профили Anomaly обходят 32-битный `AnomalyLauncher.exe` и напрямую запускают выбранный 64-битный `AnomalyDX*.exe`. Режим `Auto` читает `AnomalyLauncher.cfg`; в настройках профиля также доступны DX8, DX9, DX10, DX11 и AVX. Выбор хранится как относительный путь, поэтому `FileLayerPlan` по-прежнему получает EXE из включённого мода с наивысшим приоритетом.

## Текущая интеграция

`UsvfsMappingPlanBuilder` преобразует существующую модель лаунчера в операции официального USVFS runtime:

```text
корень мода 1 -> виртуальный корень игры
корень мода 2 -> виртуальный корень игры
...
известные writable-файлы -> точные виртуальные пути
userdata\overwrite -> виртуальный корень игры с разрешением создавать файлы
```

Runtime-backend создаёт небольшой профильный bootstrap-корень. Выбранный EXE, его loader-time зависимости и сгенерированный `fsgame.ltx` физически находятся в нём. Профили OGSR/X-Ray используют bootstrap как виртуальный корень игры, поскольку некоторые движки определяют корень по пути загруженного модуля. Anomaly сохраняет физический корень игры как целевой каталог USVFS, потому что загрузка loose-скриптов через отдельный корень работает ненадёжно. Наложение базового каталога на самого себя намеренно исключено в обеих стратегиях: оно может скрыть физические архивы `gamedata.db*`.

Путь workspace резервируется до подготовки запуска любым backend-ом. Он закрепляется за ID профиля и содержит стандартный marker лаунчера, поэтому переименование профиля не создаёт вторую папку, а удаление использует одинаковые правила безопасности в режимах `LinkedWorkspace` и `VirtualFileSystem`.

Таким образом сохраняется один источник истины:

- `FileLayerPlan` определяет порядок слоёв;
- `OverlayManifest` определяет исполняемый файл, writable-файлы и хранилище overwrite;
- `UsvfsMappingPlanBuilder` преобразует эти слои в операции USVFS.

## Почему выбрано это направление

Удалённая самописная VFS-система пыталась решить слишком много задач нативного hooking внутри самого лаунчера. Она часто завершалась с ошибками и ненадёжно подключала слои модов.

Новое направление намеренно уже:

1. сохранить `LinkedWorkspace` как стабильный backend запуска;
2. использовать `FileLayerPlan` и `OverlayManifest` как общую логику;
3. изолировать адаптер официального USVFS за `IProfileLaunchBackend`;
4. предоставить USVFS как явно выбираемую экспериментальную настройку профиля, не меняя backend по умолчанию.

## Ручная проверка

1. Поместить рядом с лаунчером `usvfs_x64.dll`, `usvfs_proxy_x64.exe`, `usvfs_x86.dll` и `StalkerModLauncher.UsvfsX86Host.exe`.
2. Открыть настройки профиля и выбрать `USVFS — экспериментальный`.
3. Для Anomaly оставить `Auto` либо выбрать нужный renderer.
4. Запустить профиль и проверить файлы модов, сохранения, настройки и журнал приложения.
5. Если игровая сборка несовместима с текущим runtime USVFS, вернуть профиль в режим `Workspace — стабильный`.

## Важные ограничения

- Не заменять `LinkedWorkspace`, пока USVFS остаётся экспериментальным.
- Не возвращать самописные hook DLL.
- Не позволять USVFS записывать в папки игры или модов.
- Все новые и изменённые файлы должны направляться в управляемое writable-хранилище профиля.
