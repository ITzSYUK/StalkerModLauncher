# Техническая документация

Документ описывает архитектуру S.T.A.L.K.E.R. Mod Launcher `v1.2.0`, хранение данных, режимы запуска и ограничения безопасности.

## Назначение и совместимость

Лаунчер работает с локальными папками игр и модификаций. Он не зависит от Steam, облачных сервисов или базы данных.

Главный сценарий: базовая DRM-free установка одной из игр серии S.T.A.L.K.E.R. и набор папок модов, которые нужно наложить друг на друга. Файловая модель применима к Shadow of Chernobyl, Clear Sky, Call of Pripyat, Anomaly, OGSR и другим сборкам с типичной структурой X-Ray: `gamedata`, `fsgame.ltx`, `bin` или `bin_x64`, архивы `gamedata.db*`.

Лаунчер не подменяет установщик мода. Пользователь сам указывает готовые папки игры и модификаций.

## Профили и порядок модов

Профиль хранит:

- имя, тип и состояние;
- путь к базовой игре;
- включённые моды и их порядок;
- выбранный EXE, источник EXE, аргументы и рабочую папку;
- режим запуска;
- закреплённый путь workspace;
- профильные настройки Discord и Anomaly USVFS.

Есть два типа профилей:

- **Обычный профиль** накладывает моды поверх базовой игры.
- **Автономный профиль** запускает готовую сборку из её собственной папки без workspace и наложения слоёв.

Слои обычного профиля применяются в таком порядке:

```text
базовая игра -> мод 1 -> мод 2 -> ... -> профильные writable-файлы
```

Мод ниже в списке имеет больший приоритет. Если два слоя содержат один относительный путь, игра получает файл из последнего включённого слоя.

## Единая модель наложения

Оба backend-а используют одну модель и не вычисляют порядок файлов независимо друг от друга.

### FileLayerPlan

`FileLayerPlan` описывает базовую игру, включённые моды и их приоритет. Он умеет определить:

- итоговый источник относительного файла;
- все слои, содержащие файл;
- кандидатов на запуск;
- файлы, которые один мод заменяет у предыдущих слоёв.

### OverlayManifest

`OverlayManifest` — снимок важных результатов плана: выбранный EXE, системные файлы, writable-зоны и источники. Он используется при подготовке запуска и диагностике.

### LaunchPlan

`LaunchPlan` — окончательный план запуска. Он содержит backend, абсолютный путь к EXE, рабочую папку, аргументы и диагностическое описание источника бинарника. `ProfileLauncher` запускает только подготовленный план.

## Режимы запуска

### Workspace — стабильный

`LinkedWorkspace` является режимом по умолчанию. Для обычного профиля создаётся управляемая структура:

```text
<корень Workspaces>\<имя при создании>-<короткий ID>\
  .stalker-launcher-workspace
  current\
  userdata\
  build-manifest.json
```

- `current` — восстановимый кэш итоговой игры;
- `userdata` — сохранения, настройки, логи, скриншоты и writable-файлы профиля;
- `build-manifest.json` — подпись источников и статистика последней сборки;
- `.stalker-launcher-workspace` — защитный marker управляемой папки.

На одном NTFS-томе исходные файлы обычно подключаются hard link. Между томами используются symbolic link. Изменяемые файлы, например `fsgame.ltx`, создаются локально. Лаунчер не переходит к скрытому полному копированию игры: если безопасную ссылку создать нельзя, запуск останавливается с понятной ошибкой.

Для symbolic link Windows может потребовать **Режим разработчика** или запуск лаунчера от имени администратора. Проводник считает логический размер hard-linked файлов как отдельный объём, хотя дополнительные блоки данных на диске не создаются.

Workspace закрепляется за ID профиля. Переименование профиля не меняет путь уже созданной папки.

### USVFS — экспериментальный

`VirtualFileSystem` использует официальный [ModOrganizer2/usvfs](https://github.com/ModOrganizer2/usvfs). Файлы игры и модов становятся видимыми только запущенному процессу без физической сборки `current`.

USVFS не является режимом по умолчанию. Для несовместимой или нестандартной сборки пользователь может в любой момент вернуться к стабильному Workspace.

Runtime рядом с лаунчером:

```text
usvfs_x64.dll
usvfs_proxy_x64.exe
usvfs_x86.dll
StalkerModLauncher.UsvfsX86Host.exe
```

x64-игры запускаются через managed-адаптер и `usvfs_x64.dll`. x86-игры запускает отдельный 32-битный helper, загружающий `usvfs_x86.dll`; helper остаётся активным, пока работают дочерние процессы игры. Одновременно разрешён один USVFS-профиль, чтобы изолировать глобальное состояние runtime.

Для сборок, где движок предоставлен модом, создаётся небольшой `userdata\usvfs-bootstrap`: там физически находятся выбранный EXE, необходимые loader-time DLL и профильный `fsgame.ltx`. Если запускается бинарник базовой игры, USVFS может использовать физический корень игры без bootstrap-копии движка.

Anomaly Launcher является 32-битной оболочкой, поэтому USVFS запускает выбранный `AnomalyDX*.exe` напрямую. В режиме **Авто** renderer читается из `AnomalyLauncher.cfg`; также доступны DX8/DX9/DX10/DX11 и AVX. Относительный EXE разрешается через `FileLayerPlan`, поэтому бинарник мода с более высоким приоритетом заменяет одноимённый базовый файл.

Глубокие исследовательские заметки и PoC находятся в [USVFS_RESEARCH.md](USVFS_RESEARCH.md).

## Профильные данные и fsgame.ltx

Для обычного профиля `fsgame.ltx` берётся из итогового слоя и переписывается так, чтобы `$app_data_root$` указывал на профильный `userdata`. Остальные строки и дополнительные алиасы мода сохраняются. Кодировка исходного файла определяется и сохраняется, включая Windows-1251.

Типичные каталоги:

```text
userdata\savedgames
userdata\logs
userdata\screenshots
userdata\user.ltx
userdata\writable-game-files
userdata\overwrite
```

`writable-game-files` используется для известных конфигураций, которые движок ожидает внутри игрового дерева, например Anomaly `gamedata\configs\localization.ltx`. `overwrite` предназначен для новых или изменённых файлов USVFS-профиля. Исходные папки игры и модов остаются источниками только для чтения.

Существующий `user.ltx` профиля не перезаписывается. При первом запуске лаунчер может импортировать исходный `user.ltx`, после чего профильная копия становится независимой.

## Безопасность файлов

Лаунчер не редактирует и не удаляет исходные папки игры или модов.

Marker-файл `.stalker-launcher-workspace` содержит идентификатор управляемой рабочей среды. Перед очисткой, пересборкой или удалением лаунчер проверяет marker и допустимый корень Workspaces. Если marker отсутствует или не соответствует профилю, операция удаления блокируется. Это защищает произвольную пользовательскую папку от ошибочной рекурсивной очистки.

Удаление обычного профиля удаляет только его управляемый workspace. Удаление автономного профиля удаляет запись настроек, но сохраняет папку готовой сборки.

## Настройки

Настройки хранятся отдельно от игры:

```text
%APPDATA%\StalkerModLauncher\settings.json
%APPDATA%\StalkerModLauncher\settings.backup.json
```

Текущая `SchemaVersion` — `4`. При чтении нормализатор заполняет отсутствующие поля, исправляет повторяющиеся ID и сбрасывает временное состояние `IsRunning`. Сохранение выполняется через временный файл и замену основного JSON; резервная копия помогает восстановиться после прерванной записи.

Пути абсолютные. После переноса игры или модов на другой диск их нужно исправить в профиле. Workspace переносится штатной кнопкой, которая обновляет путь только после успешной операции.

Лаунчер запрещает одновременную работу двух экземпляров, чтобы они не записывали один `settings.json` параллельно.

## Проверка и диагностика

Перед запуском выполняется preflight:

- проверка базовой игры и включённых модов;
- разрешение итогового EXE через общую модель слоёв;
- проверка рабочей папки и архитектуры EXE;
- наличие требуемого USVFS runtime;
- подготовка профильных writable-файлов;
- проверка безопасности workspace.

Окно **Состояние** показывает компактный итог: готовность, размер и типы файлов workspace, проверки, последний лог и crash dump. Журнал приложения записывает backend, источник EXE, причины пересборки, статистику ссылок и результат запуска.

Журнал ротируется и не растёт бесконечно. Старые записи ограничиваются настройками сервиса логирования.

## Браузер модификаций AP-PRO

Браузер читает публичные страницы категорий AP-PRO для Shadow of Chernobyl, Clear Sky и Call of Pripyat. Карточка содержит название, краткое описание, обложку, оценку, просмотры и ссылку на исходную страницу.

Страницы подгружаются последовательно при прокрутке. Загрузка категории отменяется при переключении, поэтому устаревший ответ не может заменить новый список. Поиск работает по уже полученным названиям и продолжает учитывать результаты следующих страниц. Сообщение «ничего не найдено» показывается только после завершения загрузки.

Ответы каталога кэшируются в памяти примерно на 10 минут; обложки декодируются лениво. После закрытия окна постоянный каталог на диск не сохраняется. Нужен интернет. Лаунчер не обходит защиту сайта, не скачивает и не устанавливает моды.

## Сборка и тесты

Для разработки нужен .NET 8 SDK:

```powershell
dotnet build .\StalkerModLauncher.sln -c Release
dotnet test .\StalkerModLauncher.sln -c Release
dotnet run --project .\src\StalkerModLauncher\StalkerModLauncher.csproj
```

Релиз `v1.2.0` собирается скриптом:

```powershell
.\scripts\Build-Release.ps1 -Version 1.2.0
```

Скрипт создаёт две ZIP-поставки в `publish\release\v1.2.0`:

- framework-dependent: нужен .NET 8 Desktop Runtime x64;
- standalone: .NET включён в `StalkerModLauncher-Standalone.exe`.

Обе поставки содержат необходимые native-файлы USVFS, `LICENSE.txt` и `THIRD-PARTY-NOTICES.txt`. README и release notes в пользовательский ZIP не дублируются. Для воспроизводимой упаковки локально должны быть собраны официальный USVFS x64/x86 и x86 helper; эти бинарные артефакты не хранятся в Git.

## Известные ограничения v1.2.0

- USVFS остаётся экспериментальным; нестандартные wrappers и отдельные движки могут требовать Workspace.
- Для USVFS нужен Microsoft Visual C++ 2015-2022 Redistributable соответствующей архитектуры.
- Symbolic link между дисками зависит от настроек Windows.
- Абсолютные пути не переживают перенос игры автоматически.
- Браузер модификаций зависит от доступности и разметки AP-PRO.
