# Техническая документация

[Русская версия](TECHNICAL_RU.md) | [English version](TECHNICAL_EN.md) | [Руководство пользователя](USER_GUIDE_RU.md)

Документ описывает устройство S.T.A.L.K.E.R. Mod Launcher `v1.2.2`: как хранятся профили, как определяется итоговый файл, чем отличаются Workspace и USVFS, где находятся пользовательские данные и какие проверки защищают исходные папки.

Подробная история исследования USVFS и экспериментальные прототипы находятся в [USVFS_RESEARCH_RU.md](USVFS_RESEARCH_RU.md).

## 1. Назначение и границы проекта

Лаунчер работает с локальными папками игр и модификаций. Основной сценарий состоит из базовой игры и упорядоченного набора папок, содержимое которых нужно наложить поверх неё.

Поддержка не привязана к названию конкретной игры. Файловая модель рассчитана на проекты семейства X-Ray с привычными элементами:

- `gamedata`;
- `fsgame.ltx`;
- `bin` или `bin_x64`;
- архивы `gamedata.db*`, а также архивы в `db` и `patches`;
- `appdata`, `userdata` или `_appdata_`.

На практике это охватывает Shadow of Chernobyl, Clear Sky, Call of Pripyat, Anomaly, OGSR, iX-Ray и многие производные сборки. Совместимость с каждой необычной оболочкой или модифицированным движком не гарантируется.

Лаунчер не является установщиком модов. Он не распаковывает архивы и не исправляет несовместимый игровой контент.

## 2. Основные понятия

### Профиль

В настройках каждый профиль хранится как отдельный рецепт запуска. В нём находятся:

- внутренний уникальный ID и отображаемое имя;
- тип профиля;
- путь к базовой игре;
- список модов, их порядок и включённое состояние;
- режим запуска;
- относительный путь к EXE и, при ручном выборе, закреплённый источник;
- аргументы и относительную рабочую папку;
- путь workspace;
- время игры, дату последнего запуска, Discord и выбранный графический режим Anomaly.

Вычисляемые данные вроде текущего состояния «запущен», форматированного времени и даты в JSON не записываются.

### Обычный профиль

Обычный профиль строит набор слоёв:

```text
базовая игра -> мод 1 -> мод 2 -> ... -> изменяемые данные профиля
```

Мод ниже в интерфейсе имеет больший приоритет. При совпадении относительного пути побеждает последний включённый слой.

### Автономный профиль

Автономный профиль запускает готовую сборку из её собственной папки. Для него не создаются план наложения, `current` и отдельная сборка модов. Путь к сохранениям и логам определяется самой сборкой через `fsgame.ltx` и типичные каталоги данных.

## 3. Общий путь запуска

Запуск проходит одну последовательность независимо от выбранного режима:

1. Проверяются папки игры и модов, EXE, рабочая папка и доступность выбранного режима.
2. За ID обычного профиля закрепляется его управляемая рабочая папка.
3. Строится единый план слоёв и определяется, какой файл побеждает в каждом совпадающем пути.
4. Workspace или USVFS подготавливает окончательные параметры запуска.
5. Игра запускается с выбранным EXE, аргументами и рабочей папкой.
6. Лаунчер отслеживает завершение процесса, время игры и Discord Rich Presence.
7. После выхода ищутся свежий игровой лог и аварийный дамп.

Результатом подготовки является небольшая инструкция: какой режим использовать, какой абсолютный EXE запустить, какие аргументы передать и какую рабочую папку назначить. Для USVFS в неё также входит временный сеанс виртуальной файловой системы.

Благодаря общей инструкции окно «Состояние» и реальный запуск используют один и тот же EXE и одинаковый порядок модов.

## 4. Единая модель слоёв

### План слоёв файлов

Перед проверкой или запуском обычного профиля лаунчер создаёт единый план наложения. Он содержит:

- базовую игру с порядком `0`;
- только включённые моды в пользовательском порядке;
- `userdata` как последний слой профильных данных.

С помощью этого плана лаунчер отвечает на четыре основных вопроса:

- какой файл с указанным относительным путём в итоге увидит игра;
- какие слои предоставляют этот файл;
- какие EXE доступны и откуда они взяты;
- какие файлы один мод перекрывает в предыдущих слоях.

Workspace и USVFS получают один план и не реализуют собственную трактовку приоритетов.

### Снимок итогового наложения

После анализа создаётся компактный снимок важных результатов:

- слои и их порядок;
- выбранный EXE и его источник;
- `fsgame.ltx`, `user.ltx`, `localization.ltx` и другие важные файлы;
- файлы, которые игра должна иметь возможность изменять;
- корень `userdata\overwrite`;
- при необходимости список перекрытий.

Этот снимок в памяти не следует путать с `build-manifest.json` Workspace. Первый описывает то, что должно получиться, а второй хранит сведения о последней физической сборке `current`.

## 5. Разрешение EXE

Автоматический поиск рассматривает кандидатов из базовой игры и модов. При одинаковом относительном пути выбирается файл самого приоритетного слоя.

Ручной выбор сохраняет две части:

- относительный путь к EXE внутри игрового дерева;
- закреплённую папку, из которой должен быть взят именно этот EXE.

Это позволяет намеренно выбрать бинарник нижнего слоя, даже если более приоритетный мод содержит одноимённый EXE. Кнопка автоматического выбора очищает закреплённый источник.

Для Anomaly USVFS поддерживает два сценария:

- автоматический режим: `AnomalyLauncher.exe` запускается через вспомогательную 32-битную программу, а созданный им игровой процесс наследует USVFS;
- ручной графический режим: напрямую запускается выбранный `AnomalyDX*.exe`, включая AVX-вариант.

Если EXE выбранного графического режима с тем же относительным путём предоставлен модом, используется файл из самого приоритетного слоя.

## 6. Workspace: стабильный режим

Стабильный режим собирает для профиля управляемую рабочую папку. По умолчанию она создаётся на диске базовой игры:

```text
<диск игры>\StalkerModLauncher\Workspaces\<имя>-<короткий ID>\
```

Если диск определить нельзя, резервный корень находится в:

```text
%LOCALAPPDATA%\StalkerModLauncher\Workspaces
```

Структура:

```text
<workspace>\
  .stalker-launcher-workspace
  build-manifest.json
  current\
  userdata\
```

### Подготовка current

Перед сборкой лаунчер получает снимок базовой игры и модов, затем строит подпись из путей, состояния файлов, порядка модов, выбранного EXE и параметров профиля.

Если подпись совпадает с `build-manifest.json` и выбранный EXE существует, `current` используется повторно. Изменяемые файлы профиля всё равно восстанавливаются и проверяются.

При изменении источников выполняется пересборка:

1. сохраняются известные изменяемые файлы из старого `current`;
2. содержимое `current` безопасно удаляется;
3. подключается базовая игра;
4. по порядку применяются включённые моды;
5. создаются профильные `fsgame.ltx`, `user.ltx` и другие изменяемые файлы;
6. при ручном выборе закрепляется нужный источник EXE;
7. записывается новый `build-manifest.json`.

### Стратегия файлов

Для каждого файла лаунчер выбирает подходящий способ размещения:

- жёсткая ссылка (hard link), когда источник и workspace находятся на одном NTFS-томе;
- символическая ссылка (symbolic link), когда требуется ссылка между томами;
- локальный файл для конфигурации, которая должна изменяться независимо;
- отдельная локальная копия с запретом записи только там, где ссылка небезопасна для исходника.

Скрытого перехода к полному копированию игры нет. Если безопасную ссылку создать нельзя, подготовка останавливается с ошибкой.

Жёсткая ссылка разделяет физические блоки с исходным файлом, поэтому запись через неё изменила бы источник. По этой причине потенциально изменяемые файлы не оставляются обычными доступными для записи жёсткими ссылками. Атрибуты и локальные копии защищают исходные папки.

Проводник считает размер файлов, подключённых жёсткими ссылками, как обычный размер папки. Фактически эти данные не записываются на диск второй раз. Реальное дополнительное место считается только по локальным файлам и отображается отдельно.

## 7. USVFS: экспериментальный режим

Режим USVFS использует официальную библиотеку [ModOrganizer2/usvfs](https://github.com/ModOrganizer2/usvfs). Вместо `current` она показывает запущенной игре единое виртуальное дерево файлов.

Общий план слоёв преобразуется в такую последовательность подключений:

```text
базовая игра -> моды по порядку -> известные изменяемые файлы -> папка накопленных изменений
```

Папка `userdata\overwrite` подключается последней, принимает накопленные изменения и имеет наивысший приоритет.

### x64 и x86

x64 означает 64-битную игру, а x86 - 32-битную.

x64-игры запускаются через официальную библиотеку `usvfs_x64.dll`. Для x86 используется отдельная вспомогательная программа `StalkerModLauncher.UsvfsX86Host.exe`, которая загружает `usvfs_x86.dll` в 32-битном процессе.

Вспомогательная x86-программа остаётся активной, пока работают созданные через неё игровые процессы. Это необходимо для программ запуска, которые быстро закрываются после старта движка.

Одновременно допускается одна USVFS-сессия, поскольку официальная библиотека использует общие служебные данные и единое пространство имён.

### Выбор виртуального корня

Лаунчер выбирает способ создания виртуального корня по типу запуска:

- физический корень базовой игры используется, когда запускается её собственный EXE и моды не предоставляют файлы, необходимые Windows ещё до старта движка;
- физический корень Anomaly используется для надёжной загрузки распакованных ресурсов;
- физический корень X-Ray 1.6 используется при наличии `$arch_dir_*`, чтобы не нарушать поиск архивов в `patches` и других каталогах;
- изолированная стартовая папка используется, когда движок предоставлен модом или ему требуется собственный комплект DLL.

Наложение физической папки игры самой на себя запрещено: такая операция может скрыть реальные архивы `gamedata.db*`.

### Стартовая папка usvfs-bootstrap

`userdata\usvfs-bootstrap` содержит только файлы, которые Windows и движок должны увидеть физически ещё до полноценного виртуального доступа:

- выбранный EXE;
- DLL, которые должны находиться рядом с EXE уже в момент запуска;
- профильный `fsgame.ltx`;
- минимально необходимые соседние файлы.

Это служебный кэш, а не копия всей игры. При переносе workspace эта стартовая папка не считается ценным пользовательским содержимым и создаётся заново.

Служебные файлы USVFS рядом с лаунчером:

```text
usvfs_x64.dll
usvfs_proxy_x64.exe
usvfs_x86.dll
usvfs_proxy_x86.exe
StalkerModLauncher.UsvfsX86Host.exe
```

## 8. Изоляция пользовательских данных

Для обычного профиля постоянные данные находятся в:

```text
<workspace>\userdata
```

Лаунчер берёт итоговый `fsgame.ltx`, сохраняет его кодировку, включая Windows-1251, и заменяет `$app_data_root$` на абсолютный путь профильного `userdata`. Остальные правила путей и дополнительные строки модификации сохраняются.

Типичные данные:

```text
userdata\savedgames
userdata\logs
userdata\screenshots
userdata\user.ltx
userdata\shaders_cache
userdata\writable-game-files
userdata\overwrite
userdata\usvfs-bootstrap
```

### user.ltx

При первом запуске источник выбирается от самого приоритетного слоя к базовой игре. Если профильный файл уже изменён и не совпадает ни с одним нижним источником, он считается пользовательским и сохраняется.

Если текущая профильная копия всё ещё равна файлу старого нижнего слоя, новый файл из более приоритетного патча может безопасно заменить её.

### Кэш шейдеров

Готовый кэш шейдеров собирается из appdata-папок всех слоёв с учётом приоритета. Уже существующие пользовательские файлы не затираются без необходимости. Это поддерживает сборки Anomaly, которым вместе с модами поставляется предварительно подготовленный кэш.

### Изменяемые файлы внутри игрового дерева

Некоторые движки записывают конфигурацию внутрь игрового дерева. Для известных путей, например `gamedata\configs\localization.ltx`, хранится независимая профильная копия в `userdata\writable-game-files`.

Workspace временно подставляет её в `current` и забирает изменения обратно. USVFS отображает её по нужному виртуальному пути.

### Автономные профили

Для автономного профиля лаунчер проверяет путь из `fsgame.ltx` и типичные расположения `appdata`, `userdata`, `_appdata_`, `bin\_appdata_` и `bin_x64\_appdata_`. Маршрутизация данных самой сборки при этом не меняется.

## 9. Безопасность файлов и жизненный цикл workspace

Исходные папки игры и модов не удаляются при удалении профиля.

Управляемый workspace защищён двумя уровнями:

- корневая служебная метка `.stalker-launcher-workspace-root` подтверждает допустимый каталог Workspaces;
- профильная служебная метка `.stalker-launcher-workspace` связывает папку с ID профиля.

Перед рекурсивной очисткой, переносом или удалением проверяются:

- нахождение внутри разрешённого корня;
- наличие защитной служебной метки;
- соответствие короткого ID;
- отсутствие опасного вложения исходника и назначения.

Если проверка не проходит, операция блокируется. Для автоматически созданной папки служебная метка может быть восстановлена только при однозначном совпадении с ID профиля.

Путь workspace закрепляется за ID после первого назначения. Переименование профиля не меняет существующий путь. Копия профиля получает новый ID и отдельную папку.

Удаление обычного профиля удаляет только управляемый workspace. Удаление автономного профиля удаляет запись настроек, но сохраняет папку сборки.

## 10. Настройки и восстановление

Файлы настроек:

```text
%APPDATA%\StalkerModLauncher\settings.json
%APPDATA%\StalkerModLauncher\settings.backup.json
```

В JSON хранится номер версии структуры настроек. Сейчас он равен `4`. При чтении файла лаунчер:

- обновляет версию схемы;
- создаёт отсутствующие коллекции;
- исправляет пустые и повторяющиеся ID;
- нормализует порядок модов;
- сбрасывает временную отметку о запущенной игре;
- переносит отдельные устаревшие поля.

Настройки сохраняются через временный файл, чтобы снизить риск получить повреждённый JSON при сбое:

1. полный набор настроек преобразуется в JSON;
2. он записывается в `.tmp`;
3. основной файл заменяется;
4. предыдущая версия становится резервной копией.

Чтение и запись настроек выполняются по очереди, поэтому две операции не могут одновременно менять файл. Запуск второго экземпляра приложения также блокируется.

Все игровые и модовые пути абсолютные. После переноса источника его нужно выбрать заново. Встроенный перенос workspace сначала копирует `userdata`, обновляет путь только после успеха и затем удаляет старую управляемую папку.

## 11. Проверка, состояние и диагностика

Проверка перед запуском контролирует:

- существование базовой игры и всех включённых модов;
- итоговый EXE и его разрядность, 32 или 64 бита;
- рабочую папку и аргументы;
- наличие служебных файлов USVFS нужной разрядности;
- готовность `fsgame.ltx` и профильных данных;
- безопасность workspace;
- наличие типичных DLL, которые должны лежать рядом с выбранным движком уже при его запуске.

Ошибки блокируют запуск. Предупреждения сообщают о необычной конфигурации, но могут допускать продолжение.

Окно «Состояние» использует тот же план слоёв и показывает краткую сводку. Для Workspace статистика берётся из `build-manifest.json` без повторного чтения всех файлов. Для USVFS показываются слои и готовность профильных данных, поскольку `current` отсутствует.

Логи приложения:

```text
%APPDATA%\StalkerModLauncher\launcher.log
%APPDATA%\StalkerModLauncher\launcher.old.log
```

При достижении 1 МБ текущий журнал переименовывается в `launcher.old.log`, а запись начинается в новом файле. Старая резервная версия журнала заменяется.

Игровые логи и аварийные дампы ищутся в профильном `userdata` или типичных appdata-папках автономной сборки. При поиске учитывается время запуска, поэтому старый дамп не выдаётся за результат текущей сессии.

## 12. Управление модами

Сканирование рекурсивно ищет корни модов, но после обнаружения корректного корня не добавляет его внутренние папки как отдельные моды. Распознаются обычные распакованные файлы и X-Ray-архивы, включая каталоги `db` и `patches`.

Групповое перемещение сохраняет взаимный порядок выделенных модов. Интерфейс поддерживает перетаскивание и команды перемещения в начало или конец.

Импорт из `modlist.txt` сопоставляет записи с уже добавленными модами и переносит их включённое состояние и порядок. Формат MO2 преобразуется в правило лаунчера «ниже означает больший приоритет».

Анализ перекрытий считает замену файлов информацией о порядке, а не ошибкой. Сам факт замены файла не блокирует отключение мода.

## 13. Браузер модификаций AP-PRO

Браузер читает публичные категории Shadow of Chernobyl, Clear Sky и Call of Pripyat. Он не скачивает и не устанавливает модификации.

При работе с сайтом лаунчер соблюдает следующие правила:

- честно указывает своё имя и версию в `User-Agent`, то есть в служебной подписи сетевого запроса;
- последовательная загрузка страниц с небольшой задержкой;
- не более четырёх одновременных загрузок обложек;
- если сайт отвечает кодом `429 Too Many Requests`, лаунчер ждёт указанное сервером время, но не более 30 секунд, и делает только одну повторную попытку;
- отмена старой загрузки при переключении категории;
- хранение уже загруженного списка в оперативной памяти примерно 10 минут;
- обработка изображения только в тот момент, когда оно действительно требуется интерфейсу.

Поиск фильтрует полученные названия и продолжает учитывать новые страницы. Сообщение об отсутствии результата появляется только после завершения загрузки категории.

Зависимость от HTML-разметки AP-PRO остаётся внешним риском.

## 14. Дополнительные возможности

- Поиск скриншотов находит PNG, JPG и BMP в профильных и автономных каталогах.
- Копирование скриншота освобождает полноразмерное изображение после передачи в буфер обмена и не удерживает его в памяти без необходимости.
- Discord Rich Presence публикует имя профиля и состояние запуска только при включённой настройке.
- Проверка обновлений выполняется только вручную и сравнивает версию программы с тегом последнего релиза на GitHub.
- Звуки интерфейса воспроизводятся повторно без создания нового декодера при каждом нажатии.
- Открытие дочерних окон вынесено из главного окна, чтобы его код отвечал только за отображение и пользовательские действия.

## 15. Структура приложения

```text
src/StalkerModLauncher/
  Models/          сохраняемые данные и итоговые параметры запуска
  ViewModels/      состояние экранов и пользовательские команды
  Views/           окна и переиспользуемые элементы интерфейса
  Themes/          палитра и стили
  Services/        основная логика запуска, workspace, USVFS и AP-PRO
  Infrastructure/  базовые элементы интерфейса и команды

native/
  StalkerModLauncher.UsvfsX86Host/

research/
  usvfs-poc/
  usvfs-managed-poc/

tests/StalkerModLauncher.Tests/
```

При запуске приложения один общий модуль создаёт необходимые компоненты. Workspace доступен всегда, а USVFS подключается только при наличии его служебных файлов или включённом специальном тестовом флаге.

В коде интерфейса отдельно хранятся отображение, состояние и пользовательские команды. Логика главного экрана разделена по сценариям, а крупные части окна вынесены в отдельные элементы управления.

## 16. Сборка, тесты и релиз

Для разработки нужен .NET 8 SDK:

```powershell
dotnet build .\StalkerModLauncher.sln -c Release
dotnet test .\StalkerModLauncher.sln -c Release
dotnet run --project .\src\StalkerModLauncher\StalkerModLauncher.csproj
```

Полная релизная упаковка:

```powershell
.\scripts\Build-Release.ps1 -Version 1.2.2
```

Создаются два ZIP:

- компактный пакет, которому нужен установленный .NET 8 Desktop Runtime x64;
- автономный standalone-пакет со встроенным .NET.

Оба варианта содержат официальные файлы USVFS x64/x86, вспомогательную x86-программу, `LICENSE.txt` и `THIRD-PARTY-NOTICES.txt`. PDB, JSON, Markdown и промежуточные файлы в пользовательские ZIP не включаются.

Тестовая VFS-сборка:

```powershell
.\scripts\Build-VfsExperimental.ps1 -CleanPublishRoot
```

Собранные служебные файлы официального USVFS и вспомогательная x86-программа должны быть подготовлены локально. Скомпилированные сторонние бинарники в Git не хранятся.

## 17. Известные ограничения

- USVFS остаётся экспериментальным; для несовместимой сборки используется Workspace.
- Для USVFS нужен Microsoft Visual C++ 2015-2022 Redistributable архитектуры запускаемой игры.
- Символические ссылки между дисками зависят от настроек Windows.
- Абсолютные пути не исправляются автоматически после переноса игры или модов.
- Автономный профиль не может гарантировать отдельные сохранения, если сама сборка использует общую внешнюю папку.
- Автоматическое определение EXE и корня мода не заменяет требования автора конкретной сборки.
- Браузер модификаций зависит от доступности и HTML-разметки AP-PRO.
